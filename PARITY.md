# Parity with `foundation-registry`

This fork tracks `kaelys-js/foundation-registry` for the shared toolchain
(config, qa, sync) and diverges on top for the `claude-multiacct` runtime
product. This file records every file that intentionally differs between the
two working trees, and why.

This is the fork's first commit (rebuild PR0). The repo previously held a Bash
tool for running multiple Claude Desktop instances; that tree was replaced
wholesale with a foundation-registry fork. The product package is a scaffold
only тАФ no product logic yet.

Reference diff command (from either checkout root):

```sh
diff -rq /path/to/foundation-registry /path/to/claude-multiacct \
  --exclude=.git --exclude=node_modules --exclude=coverage \
  --exclude=.turbo --exclude=.mise \
  --exclude=packages/products/claude-multiacct \
  --exclude=.github/workflows/sync-upstream.yml \
  --exclude=.sync-upstream.json \
  --exclude=PARITY.md --exclude=PARITY.md.template
```

## Intentional drifts

### Product set

- `packages/products/claude-multiacct/` тАФ fork-only. The single-app OAuth
  account-pool product; the sole reason this fork exists. Scaffold only in
  this PR (an exported package-identity constant plus its intent test); the
  runtime lands in later PRs.
- `packages/products/registry/` тАФ upstream-only. Governance records live
  upstream; the fork does not carry them.
- `packages/products/trp/` тАФ upstream-only. The task-runner product is not
  part of this fork.

### Root-level files

- `PARITY.md` тАФ this file (fork-only). Upstream ships the unfilled
  `PARITY.md.template`; the fork replaces it with this filled record.
- `package.json` тАФ root `name`/`author`/`homepage` stay as upstream's for
  now (the fork has not rescoped the root manifest). `pnpm-lock.yaml` is the
  only manifest-adjacent file that diverges, because the product set differs.

### Toolchain configs

- `pnpm-lock.yaml` тАФ different transitive graph: the fork drops the `registry`
  and `trp` products and adds `claude-multiacct`, so the resolved workspace
  set (and its dependency closure) differs from upstream.
- `vitest.config.ts` тАФ byte-identical to upstream. Its per-file coverage
  exclusion for `packages/products/trp/src/scripts/fix-task.ts` is a dead path
  here (trp is absent); excluding a non-existent path is a no-op, so the file
  is left unchanged to minimise drift.
- `.github/workflows/ci.yml` тАФ byte-identical to upstream. The `validate` job
  self-skips on this fork (no `packages/products/registry/`) via the `detect`
  job's output, so no edit is needed.

### CI + workflows

- `.github/workflows/sync-upstream.yml` тАФ fork-only. Daily upstream-merge
  workflow with auto-PR + auto-merge fallback. Generated by
  `pnpm sync-upstream:opt-in` from the `same-owner` template in
  `packages/shared/config/workflows/`, driven by `.sync-upstream.json`.

## Remotes on the local fork checkout

- `origin` тЖТ `git@github.com:kaelys-js/claude-multiacct.git`
- `upstream` тЖТ `git@github.com:kaelys-js/foundation-registry.git`

Both remotes assume SSH; a non-interactive session without the key added to
`ssh-agent` will need `GH_TOKEN`-backed HTTPS pushes as a per-session
workaround.

## Verification of the `sync-upstream.yml` workflow

The workflow's shell logic was not exercised end-to-end locally: running it
requires the `GH_TOKEN` repository secret on
`kaelys-js/claude-multiacct`, plus a non-fast-forward upstream state to reach
the merge path. The generated file is parseable YAML and schema-valid against
`.schemas/github-workflow.json` (checked by `pnpm lint:schema`). End-to-end
confirmation happens the first time the secret is set and the daily cron runs
(or via `workflow_dispatch`).
